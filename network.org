* Overview and Core Principles
This document outlines the network architecture for the Ngaruma School deployment in Tanzania.

** Core Principles applied:
- *Keep it simple, stupid! (KISS)*: IPv6 is completely disabled to reduce overhead and troubleshooting complexity. Security relies on simple, robust Layer-2 isolation and firewall rules rather than complex routing.
- *High-Performance Layer-2 Design*: To prevent routing bottlenecks, all heavy internal traffic (PXE booting 20 Thin Clients, NFS mounts, RDP sessions to Proxmox) is kept locally on the main switch in the new building. The router only handles internet traffic and DNS.
- *Physical Resilience*: The ~400m connection between the main building (Router) and the new building (Proxmox/Thin Clients) will be established using a fiber optic link (SFP+) to ensure complete galvanic isolation and immunity against lightning strikes.

* IP Addressing & Subnet Architecture
The network is divided into four distinct logical zones. VLANs are only used where technically necessary to transport multiple networks over a single trunk cable.

| Subnet         | Name            | VLAN ID | Interface / Port            | Purpose & Access Status                                  |
|----------------+-----------------+---------+-----------------------------+--------------------------------------------------|
| 172.20.0.0/24  | Admin & Mgmt    | 1 (unt) | Bridge / ether2-4 / wifi1   | Core management, Proxmox Host, Full access.      |
| 172.20.89.0/24 | Teachers        | 89      | vlan-teachers / wifi-teach  | Teacher PCs and Teacher Wi-Fi.                   |
| 172.20.90.0/24 | Students Wired  | 90      | vlan-students               | Thin Clients booting via PXE from Proxmox.       |
| 172.20.91.0/24 | Students Wi-Fi  | None    | wifi2 (Isolated)            | Isolated 2.4GHz Wi-Fi for student devices.       |

* Hardware Layout & Port Assignment

** 1. Router: MikroTik hAP ax3 (Main Building)
Acts as the central gateway, DHCP server, and firewall.

- *ether5*: WAN Uplink (DHCP Client from cellular modem).
- *ether1*: Trunk to Switch (VLAN 1 Untagged, VLAN 89 Tagged, VLAN 90 Tagged). *Note: PoE-out is strictly disabled here.*
- *ether2 - ether4*: Local Admin/Management access (Untagged, 172.20.0.x).
- *wifi1 (5GHz)*: SSID = =NgarumaAdmin=. Core management Wi-Fi.
- *wifi-teachers (5GHz Virtual)*: SSID = =NgarumaTeachers=. Mapped to VLAN 89.
- *wifi2 (2.4GHz)*: SSID = =NgarumaStudents=. Hard-removed from the bridge for Layer-2 isolation. 20MHz channel width for maximum penetration through clay walls.

** 2. Main Switch: CSS610 / CRS (New Building)
Acts as the distribution layer for the Proxmox server and Thin Clients. Configured via SwOS (Strict VLAN Mode).

| Port | Connection Target   | VLAN Mode | VLAN Receive  | Default VLAN ID | Description                                      |
|------+---------------------+-----------+---------------+-----------------+--------------------------------------------------|
| 1    | Uplink to hAP ax3   | strict    | any           | 1               | Trunk port receiving tagged and untagged traffic |
| 2    | Management/Admin PC | strict    | only untagged | 1               | Local port for switch/server management          |
| 3    | Proxmox Server (px1)| strict    | any           | 1               | Trunk for Proxmox (Host=Untagged, VMs=Tagged)    |
| 4-8  | Thin Clients        | strict    | only untagged | 90              | Access ports for student Thin Clients            |

/Note: The CSS610 serves as a temporary simulation for the planned CRS 24-Port Switch. The logic remains identical./

* Security & Infrastructure Services

** DNS Filtering
All networks are forced to use Cloudflare Family DNS (=1.1.1.3=, =1.0.0.3=) for baseline malware and adult-content filtering.

** Access Control & Isolation
- *SSH*: Password authentication is completely disabled. Ed25519 public key authentication is strictly enforced. Strong-crypto is enabled.
- *Student Wi-Fi Isolation*: A firewall forward-chain drop rule prevents any traffic originating from =172.20.91.0/24= from reaching the local LANs (=172.20.0.0/24=, =.89.0/24=, =.90.0/24=). Students only have internet access. Client isolation (station-roaming=no) is active.

** System Settings
- *Timezone*: Africa/Dar_es_Salaam (EAT).
- *NTP*: Synced against =0.pool.ntp.org= and =1.pool.ntp.org=. Router acts as local broadcast/multicast NTP server for clients.
- *Hardware LEDs*: Controlled via script bound to the physical mode-button ("dark-mode").

* Operational Notes & Maintenance
- *Emergency Access*: If locked out via VLANs, connect a laptop to =ether2= on the hAP ax3. It defaults to the native =172.20.0.x= management subnet.
- *Adding a new Teacher Wi-Fi*: Can be done by creating a new Virtual AP bound to =wifi1= and setting =datapath.vlan-id=89=.
- *Proxmox LXC Networking*: When creating a container for students (e.g., Pi-hole or specific services), set the network interface tag to =90= in the Proxmox GUI. It will receive a =172.20.90.x= IP from the MikroTik DHCP server.

* Server & Container IP Allocation
To maintain the "Keep it simple, stupid!" principle, all LXC containers and end-user devices are configured for DHCP. The MikroTik router assigns fixed IPs via static leases (=sLease=) based on MAC addresses. Only the core physical infrastructure (Router =hapax3=, Proxmox Host =px1=) retains hardcoded static IPs to prevent lockouts if the DHCP service fails.

All =sLease= IPs are assigned below the dynamic DHCP pool threshold (.100).
